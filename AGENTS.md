# AGENTS Guide for MediaCMS

## Big Picture
- MediaCMS is a Django + React media platform; Django serves templates/API and React bundles are embedded into Django pages (not a standalone SPA).
- Core backend apps are `files` (media domain + API), `users`, `actions`, `rbac`, `identity_providers`, `uploader`; see `cms/settings.py` `INSTALLED_APPS`.
- Root URL wiring is centralized in `cms/urls.py`; most site/API endpoints are in `files/urls.py`, user endpoints in `users/urls.py`.
- Main media lifecycle lives in `files/models/media.py` (`Media` model + signals) and `files/tasks.py` (Celery encoding/transcoding pipeline).
- Async boundary: web requests enqueue Celery tasks on Redis (`short_tasks` and `long_tasks` queues), workers process FFmpeg/Bento4 jobs.

## Runtime and Services
- Preferred local dev path is Docker Compose: `docker compose -f docker-compose-dev.yaml up`.
- Dev stack runs Django on `:80`, React dev server on `:8088`, plus Postgres, Redis, Celery worker, and migrations container.
- `deploy/docker/prestart.sh` runs migrations, initial fixtures, admin bootstrap, and `collectstatic`; avoid duplicating this logic in ad-hoc scripts.
- Compose/dev mode sets `DEVELOPMENT_MODE=True`, which loads `cms/dev_settings.py` (Debug Toolbar + permissive CORS).
- Production container process composition is controlled by env flags (`ENABLE_*`) in `Dockerfile` + `deploy/docker/prestart.sh`.

## Frontend/Backend Integration Pattern
- React changes are made in `frontend/src`, previewed on `http://localhost:8088`, then built and copied into Django static files.
- Use project workflow (also wrapped by `make build-frontend`):
  - `docker compose -f docker-compose-dev.yaml exec frontend npm run dist`
  - `cp -r frontend/dist/static/* static/`
  - `docker compose -f docker-compose-dev.yaml restart web`
- Template-to-bundle coupling is explicit (example: `templates/cms/media.html` loads `js/media.js`); keep bundle names stable when editing entrypoints.

## Testing and Quality Workflow
- Backend tests are expected to run with `TESTING=True` so Celery executes eagerly (`cms/settings.py` -> `CELERY_TASK_ALWAYS_EAGER`).
- Canonical command: `docker compose -f docker-compose-dev.yaml exec --env TESTING=True -T web pytest` (or `make test`).
- Frontend unit tests live under `frontend/tests` and run via `frontend/package.json` scripts (`npm run test`, `npm run test-coverage`).
- Python formatting/lint conventions come from `.pre-commit-config.yaml` (`black`, `isort`, `flake8`) and `setup.cfg` (`max-line-length=119`, `E501` ignored).

## Project-Specific Coding Conventions
- Media identifiers are usually `friendly_token` across routes and APIs; preserve this in new endpoints (`files/urls.py`, `users/urls.py`).
- `Media` side effects rely on Django signals (`post_save`, `m2m_changed`, `post_delete`) in `files/models/media.py`; prefer extending existing hooks over duplicating side effects in views.
- Encoding logic applies profile filtering and queue priorities in model/task code (`Media.encode`, `chunkize_media`, `encode_media`); avoid bypassing these by direct FFmpeg calls.
- API docs are generated by drf-yasg and exposed at `/swagger/` (`cms/urls.py`); keep serializers/views compatible with schema generation.

## External Integrations and Dependencies
- Hard dependencies: PostgreSQL, Redis, Celery, FFmpeg, Bento4, Django REST Framework, allauth/SAML (`requirements.txt`, `Dockerfile`).
- Optional transcription path uses Whisper and `requirements-full.txt`; features are gated by settings (`USE_WHISPER_TRANSCRIBE`).
- Email sending is asynchronous via `djcelery_email` and Celery queue config in `cms/settings.py`.
- File storage is local filesystem by default (`MEDIA_ROOT`/`STATIC_ROOT` in `cms/settings.py`); many workflows assume shared filesystem access for workers.
